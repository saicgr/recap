# Summarizer eval suite — 97 meeting-summary regression cases

Durable, version-controlled test set for the on-device summarizer. Each case is a
transcript we authored with **known ground truth**, so an LLM summary can be scored
mechanically. Weighted toward **long meetings** (30 long / 45 medium / 22 short) and
spread across 15 meeting types (board, incident-retro, sales, 1:1, standup, doctor
visit, legal intake, all-hands, …).

## Why synthetic, not scraped
A real transcript has no answer key. To make a summary *testable* you must plant the
content and assert on it. Every case plants distinctive facts, hallucination traps,
and continuity items, then a rubric greps the summary for them.

## Case shape (`cases.json` = array of these)
```
{ id, title, category, durationMin, lengthClass, speakers, transcript,
  rubric: {
    mustContain:    [{needle, why}]   // planted fact that MUST survive into the summary
    mustNotContain: [{needle, why}]   // hallucination trap: a plausible value NEVER said
    continuity:     [{needle, why}]   // absence/handoff/deadline planted in the last 2 min
    requiredSections:[ "TL;DR", ... ]
    lowConfidence:  [{term, why}]     // garbled term that should be FLAGGED, not asserted
  } }
```
Needles are case-insensitive substrings or regexes. Digit-joining needles (e.g.
`65766`) are validated against the *spaced* form in the transcript (`6 5 7 6 6`) —
the summary is expected to join and mark them `(heard, unverified)`.

## Provenance
Generated by a 12-agent workflow, then validated: every mustContain/continuity
needle must be recoverable from its transcript (verbatim, regex, or joined-digit
form) and every trap must be genuinely absent — otherwise the needle is pruned or
the case dropped (3 dropped for too-short transcripts). This keeps the suite from
lying about its own ground truth.

## Running (needs a local Ollama daemon + a pulled model — NOT a CI test)
```
# full suite (SLOW — hours on a 2B):
RECAP_OLLAMA_MODEL=gemma4:e2b flutter test \
  test/summarizer/eval_suite_runner_test.dart --plain-name suite
# fast subset:
RECAP_EVAL_FILTER=long   # only long meetings
RECAP_EVAL_LIMIT=8       # first N cases
```
The runner (`../eval_suite_runner_test.dart`) runs each case through the REAL
`SummaryPipeline` with the model pinned to Gemma 4 E2B's 4096-token window, scores
the rubric, and writes a report (pass rate + which needle each failure missed +
the model's actual summary) to the temp dir. That report is what the
prompt-optimization loop reads.
